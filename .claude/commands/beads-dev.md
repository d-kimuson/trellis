bdから実装可能なタスクを1つ選び、実装を完了させてください。

## 手順

### 1. タスク選択

```bash
bd ready
```

出力されたタスク一覧から、優先度が最も高いものを1つ選ぶ。

```bash
bd show <id>
```

詳細を確認し、実装方針を決める。

### 2. タスクを in_progress に更新

```bash
bd update <id> --status=in_progress
```

### 3. 実装

実装上の注意：
- コアロジックはGUI非依存に保つ
- ソースファイルを追加した場合は `Makefile` と `Package.swift` 両方を更新する
- `GhosttyKit` の型を使う場合は `import GhosttyKit`

ユニットテストで動作を保証できる箇所はt-wada流のTDDで実装する（テストを先に書き、最小実装でパスさせ、リファクタ）。GUIや ghostty C APIに依存する部分はテスト対象外。

### 4. クオリティゲート

```bash
npx -y check-changed@0.0.1-beta.4 run
```

全チェック（lint / build / test）が通過すること。失敗した場合は修正して再実行。

### 5. 動作確認依頼（条件付き）

以下の基準で動作確認依頼が必要かを判断する：

**必須**
- 機能追加
- ユーザーに見える挙動を変える bug fix

**スキップ可**
- リファクタ・内部構造の変更でテストで動作が担保されている
- bug fix だが内部実装の問題のみで、外部挙動に変化なし

スキップする場合はその理由（テストで担保済み、等）を次のステップで一言添える。

確認が必要な場合は以下のフォーマットで報告し、確認が取れるまで次のステップに進まない。

**確認手順は変更内容を踏まえて設計すること。** 起動するだけでよいのか、特定の操作が必要なのか、再起動が必要なのかを判断して具体的に記載する。デバッグログが必要な場合は `make run` の代わりに `make debug` を案内する。

### 6. コミット＆クローズ

ユーザーの動作確認OK後に実行：

```bash
bd sync --from-main
git add <変更ファイル>
git commit -m "<タスクタイトルを簡潔に>"
bd close <id>
```

### 7. 完了報告

実装内容・変更ファイル・クオリティゲート結果をまとめて報告してください。

## 制約

- 1タスクのみ実装する（複数タスクに手を出さない）
- スコープ外の修正・リファクタは行わない（別タスクを作成して記録するに留める）
- `git push` は行わない
